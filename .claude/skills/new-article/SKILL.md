# new-article

Zennに投稿する新しい記事の執筆を開始するためのスキルです。以下の手順を順番に進めてください。

## 手順

1. **作業ツリーの確認**
   - `git status` で作業ツリーがクリーンであることを確認する。未コミットの変更が残っている場合はユーザーに確認してから進める。
   - 現在のブランチが `main` でない場合、先に `main` に切り替えて最新化してよいかユーザーに確認する。

2. **タイトルを聞く**
   - ユーザーに記事のタイトルを尋ねる（自由記述の質問でよい。選択式のツールは使わない）。
   - 例: 「記事のタイトルを教えてください」

3. **概要を聞く**
   - ユーザーに記事で書きたい内容の概要を尋ねる。
   - 例: 「この記事で伝えたい内容の概要を教えてください」

4. **ブランチを作成**
   - タイトルから短い英語スラッグ（英数字とハイフンのみ、ローマ字化または意訳）を作り、`article/<slug>` という名前で新しいブランチを作成してチェックアウトする。
     ```bash
     git checkout -b article/<slug>
     ```
   - 適切な英語スラッグを作りにくい場合は日付ベース（例: `article/2026-07-22`）にフォールバックする。

5. **記事ファイルをスキャフォールド**
   - `npx zenn new:article` でファイルを生成する。スラッグはZenn側のランダム生成に任せ、指定しない（既存記事の命名規則に合わせるため）。
     ```bash
     npx zenn new:article --title "<タイトル>" --type tech --emoji "📝" --published false --machine-readable
     ```
   - 内容が明らかに技術記事でなくアイデア・意見寄りであれば `--type idea` にする。emojiは内容に合わせて変えてよい。
   - `--machine-readable` の出力から生成されたファイルパスを取得する。

6. **概要を本文に反映**
   - 生成されたMarkdownファイルの本文に、聞き取った概要をもとにした簡単な見出し構成や導入文を追加し、執筆の土台を作る。本文を最後まで書き切る必要はない。
   - 聞き取った概要は、本文の先頭に「まとめ」セクションとして記載する（例: `## まとめ` の見出しの下に概要をそのまま、または簡潔にまとめた文章を置く）。

7. **完了報告**
   - 作成したブランチ名とファイルパスをユーザーに伝える。
   - コミットやpushは行わず、ユーザーが内容を確認・編集してから明示的に依頼するのを待つ。

## 注意事項

- 既存記事ファイルのリネームやスラッグの変更は絶対に行わない（Zennの公開URLが壊れるため）。
- このskill内で生成する記事は常に `published: false` にする（`true` でデフォルトブランチにpushすると即座に公開されるため）。
- コミット・pushはユーザーから明示的に依頼されない限り行わない。
